## Context

`clone-flomo` 是 React + FastAPI 的 flomo web 端复刻项目。当前版本已经有默认备忘录、前端列表、基础 CRUD API、Docker Compose 和测试，但默认数据仍是人工样例，不能证明本地页面与 `https://v.flomoapp.com/mine` 的线上原始数据一致。

本次变更的领域边界是“备忘录工作台数据复刻”：通过线上页面分页加载获得的全部备忘录内容、标签、统计、排序和用户在工作台上完成的 CRUD 行为。真实 flomo 远程同步协议、后台定时同步和跨账号同步不在本项目领域内；但为了迁移数据，Playwright 必须使用当前已登录线上页面把可分页加载出的全部备忘录采集下来。

## Goals / Non-Goals

**Goals:**

- 使用 Playwright 对线上页面执行分页/滚动加载，直到没有更多备忘录可加载，并产出可复查截图或文本快照。
- 让本地默认数据与线上分页加载出的全部备忘录数据一致，覆盖备忘录内容、标签、排序和统计。
- 保证数据从 FastAPI 后端进入前端，不在 UI 层复制一份独立假数据。
- 用后端单元测试、前端单元测试和 E2E 测试覆盖数据一致性与完整 CRUD。

**Non-Goals:**

- 不实现真实 flomo 远程 API 同步或后台定时同步。
- 不把 Playwright 输出截图作为生产数据源提交到应用包。
- 不扩大 UI 复刻范围到当前页面之外的设置、会员、搜索高级能力。

## Decisions

### 以后端默认种子数据作为单一数据源

后端 `DEFAULT_MEMOS` 是本地默认数据的源头，前端只通过 API 读取和修改数据。这样 CRUD、Docker Compose 和 E2E 都使用同一份数据，避免前端和后端各自维护不同样例。

### Playwright 只负责采集和验收

Playwright 用于读取线上页面文本、分页加载全部备忘录、截图对比本地页面，并验证本地 CRUD 主流程。采集结果用于生成本地迁移数据和测试断言，但不引入运行时抓取逻辑，避免登录态、反爬和页面结构变化进入生产路径。

### 数据一致性覆盖分页加载出的全部数据

线上页面支持分页加载。本次验收以 Playwright 在当前登录态下通过滚动或加载按钮能够加载出的全部备忘录为准，包括统计、标签、备忘录卡片内容、时间展示和排序。采集过程必须记录最终加载数量；如果线上仍有无法通过页面加载观察的数据，必须在风险中说明，不能凭猜测补齐。

### TDD 顺序

先补后端与前端测试断言，明确线上可见数据和 CRUD 预期；再修改种子数据和 UI 展示；最后用 Playwright E2E 验证 red-green-refactor 闭环。

## Risks / Trade-offs

- 线上 flomo 页面数据会继续变化 → 本次只锁定采集时刻分页加载出的全部数据，并在 OpenSpec 与测试中记录验证范围。
- Playwright 登录态可能失效 → 若无法访问线上页，暂停数据同步并汇报，不能凭猜测补数据。
- 分页加载可能受网络或虚拟列表影响 → 使用稳定选择器和去重策略采集备忘录，记录加载前后数量，必要时多次滚动直到数量不再变化。
- 截图像素级 1:1 受字体、浏览器渲染和账号动态内容影响 → 验收以布局、文案、数据、交互和明显视觉差异为主，像素级差异作为辅助风险说明。
